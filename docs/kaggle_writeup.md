## Why this exists

Every song originates from something. In today's attention economy, the most-played songs get bumped higher — and that buries the older, path-blazing music that explains where the songs people love actually came from.

The purpose isn't just to expose listeners to that lineage, but to give them the history, context, and what was happening at the time — so they learn that nothing happens in a vacuum. This tool sits at the crossroads between delight and education, both musical and political.

Not a Wikipedia article about the Great Migration — eight songs, each introduced by a voice that tells you exactly what to listen for and why it matters. The algorithm promotes what's popular. This promotes what matters.

An agent-based system is right for this because the library needs to grow beyond what any single person could curate. The pipeline doesn't just generate content — it verifies it, finds real streaming previews, produces narration audio, and decides whether the result meets the bar. A workflow that takes a human hours per journey. The agents do it autonomously from a two-sentence request.

---

## What I built

Sonic Rabbit is a curated library of music history "podcast" journeys — narrated, chronological sequences of songs on a shared theme. Think mini audio documentary, not a playlist.

Pick a topic — gospel and soul, the Great Migration, women who rewrote music history — and the app plays: intro narration → song 1 narration → 30-second preview → song 2 narration → preview → ... → closing narration. At the end: a downloadable playlist with deep links to Spotify, Apple Music, and YouTube.

The library grows autonomously. When a user searches for something that doesn't exist, they can request it. The pipeline validates, builds, fact-checks, generates audio, and publishes — no human in the loop.

## Architecture

```
                     ┌─────────────────────────────────────────┐
                     │         CONTENT PIPELINE (build-time)   │
                     │                                         │
  theme string       │  SongPicker  →  SongFetcher (MCP)       │
  ──────────────►    │       ↓              ↓                  │
                     │  ContentWriter  ←  (confirmed songs)    │
                     │       ↓                                  │
                     │  FactChecker (MCP: Wikipedia/MusicBrainz)│
                     │       ↓                                  │
                     │  Kokoro TTS (local ONNX, no API)         │
                     │       ↓                                  │
                     │  GraphLinker → quality gate → publish   │
                     └────────────────┬────────────────────────┘
                                      │ data/journeys.json
                                      ▼
                     ┌─────────────────────────────────────────┐
                     │         WEB RUNTIME (serve-time)        │
                     │                                         │
                     │  FastAPI + Jinja2 + HTMX                │
                     │  MiniLM semantic search (local, no API) │
                     │  Request queue + rate limiter           │
                     │  Background queue worker                │
                     └─────────────────────────────────────────┘
```

The web runtime needs no Gemini API key. All content and audio is pre-generated by the pipeline and served as static files. The only runtime AI is MiniLM for semantic search — local, free, no network calls.

## The agent system

### Pipeline agents (Google ADK)

| Agent | Type | Role |
|-------|------|------|
| **SongPicker** | `LlmAgent` | Generates a ranked candidate list — intentionally over-generates so SongFetcher can cut songs without previews |
| **SongFetcher** | `LlmAgent` + MCP | Resolves each candidate: fetches `preview_url`, `image_url`, `streaming_links`. Songs without a preview are cut here |
| **ContentWriter** | `LlmAgent` | Writes intro, per-song blurbs (must name artist, year, recording location), and closing paragraph |
| **FactChecker** | `LlmAgent` + MCP | Hits MusicBrainz for recording metadata, Wikipedia for biographical claims. Flags unverifiable claims — doesn't rewrite, only annotates confidence |
| **NarrationGenerator** | Python | Kokoro-82M ONNX TTS, runs locally. Generates WAV for every narration segment. No API key, no rate limits |
| **GraphLinker** | `LlmAgent` | Reads the fully-enriched journey, counts gaps, makes a binary publish decision |

The orchestrator (`pipeline/run.py`) is deterministic Python driving all agents via ADK `Runner` with `InMemorySessionService`.

### Runtime agents (web layer)

- **Classifier**: one Gemini call per user request — spam check, category assignment, runs after MiniLM duplicate detection
- **Queue worker**: asyncio background task, polls every 60s, runs the full pipeline per approved request, reloads repo and re-indexes embedder on success

## MCP server

`src/music_journey/mcp_music/server.py` exposes three tools:

| Tool | External source | Used by |
|------|----------------|---------|
| `search_track(title, artist)` | iTunes → Deezer fallback | SongFetcher |
| `verify_recording(title, artist)` | MusicBrainz | FactChecker |
| `search_wikipedia(query)` | Wikipedia API | FactChecker |

Two hard constraints enforced through the MCP boundary:
1. `preview_url` is **never LLM-generated** — must come from a tool result or the song is cut
2. `place` (recording location) is **never LLM-generated** — must come from MusicBrainz

The pipeline cannot hallucinate a preview URL or a wrong recording location. If MusicBrainz doesn't have it, the fact checker flags it as unverified.

## Google ADK features used

**`LlmAgent` with structured output schemas** — each agent has a Pydantic schema (`pipeline/schemas.py`) constraining its return type. The orchestrator validates each output before passing it forward.

**`Runner` + `InMemorySessionService`** — each pipeline run gets its own session; agents don't share state across runs.

**MCP tool integration** — SongFetcher and FactChecker are `LlmAgent` instances with the MCP server mounted as a tool source. Agents call tools autonomously per their instruction prompts.

**Trajectory evalsets** (`tests/evals/`) — each agent has an ADK `.evalset` file testing the expected tool-call sequence and output schema. Runs in CI with `pytest`.

**Autonomous quality gate** — GraphLinker is a full `LlmAgent` (not a rule) that reads the enriched journey and decides whether to publish. Returns `publish: bool`. A deterministic fallback activates if the agent call itself fails.

## REST API

The web runtime exposes a JSON API under `/api` — consumed by the server-rendered frontend and available for any external client.

| Endpoint | Description |
|----------|-------------|
| `GET /api/journeys` | List all published journeys |
| `GET /api/journeys/{id}` | Single journey with full song list |
| `GET /api/journeys/{id}/related` | Up to N journeys ranked by category overlap |
| `GET /api/journeys/{id}/songs/{pos}` | Single song step |
| `GET /api/journeys/next?from_id=` | Next journey (category-matched, then random) |
| `GET /api/search?q=` | Semantic search — returns hit or miss + closest |
| `GET /api/chips` | Prompt chips for homepage |
| `POST /api/request-journey` | Submit a new journey request |
| `GET /health` | Liveness check with journey count |

CORS is open (`allow_origins=["*"]`). Interactive docs at `/docs` (FastAPI auto-generated OpenAPI).

## Discovery

**Rung 1 — Prompt chips**: hand-curated chips map directly to journey IDs. Zero cost, zero AI.

**Rung 2 — Semantic search**: MiniLM (all-MiniLM-L6-v2) loaded at startup. Journey titles, themes, and blurbs are embedded once. At query time: embed in ~10ms, cosine-match, threshold 0.35. Can only return journeys that exist — no hallucination.

**Rung 3 — Honest miss + request**: when nothing matches, the app shows the closest result and offers a request form. Requests go through rate limiting (3/IP/24h), MiniLM duplicate detection (≥ 0.65 = duplicate), and Gemini classifier before queuing for the pipeline.

## Quality gate

GraphLinker autonomously decides whether to publish. All must pass:

- ≥ 5 songs
- **0 missing preview URLs** — without audio, a music podcast is just text
- Narration present for at least half the songs

Failures go to `data/drafts/` with a quality report in `data/review/`. Nothing surfaces in the live library. The threshold is the threshold.

## Technical challenges

**Local TTS without API limits.** Kokoro-82M ONNX had a bug on arm64 macOS: it output int32 PCM values that sounded like noise. Fix: monkey-patch `_create_audio` to cast to float32 before the ONNX runtime. Runs at real-time speed on CPU.

**Preview URLs are the bottleneck.** About 30–40% of iTunes candidates come back without a preview URL. The pipeline over-generates (asks for 10–12, needs 7) to absorb the cuts. Songs that reach ContentWriter are guaranteed to have previews.

**Embedding drift on a small corpus.** With 6 journeys, the embedding landscape shifts each time one is added. A query scoring 0.38 with 4 journeys may score 0.34 with 6. Tests checking journey routing needed more specific queries to stay stable.

**HTMX + ADK = fully decoupled.** The web layer serves pre-generated content. HTMX handles interactive state through server-rendered fragments. ADK runs offline and writes to the same JSON the web app reads. The only shared artifact is `data/journeys.json`.

**Background worker lifecycle.** The queue worker is an asyncio task in FastAPI's lifespan context. It polls every 60s, runs the full pipeline (5–10 minutes), then reloads the repo and re-indexes the embedder in-memory — new journeys are searchable immediately without a restart.

## What I'd do differently

**Speed.** The pipeline takes 5–10 minutes. Most of that is SongFetcher calling iTunes one song at a time — the MCP tool is synchronous and the ADK runner doesn't parallelize within a single agent. Splitting into N parallel single-song agents would cut this to under 2 minutes.

**Fact-checking on obscure recordings.** MusicBrainz is reliable on mainstream catalog; for deep cuts it often returns nothing and confidence drops to `low`. The FactChecker should remove low-confidence claims from the blurb rather than just flagging them.
