# Music Journey — Product & Functional Spec (MVP)

> Status: Draft for review. This is the **WHAT**. The **HOW** (tech decisions, build
> sequence, task breakdown) lives in PLANNING.md.

---

## 1. One-line summary

A curated library of listenable music history podcasts — each **Journey** is a narrated,
chronological sequence of 6–10 songs on a shared theme, where a human voice guides the
listener through context before each song plays.

## 2. Approach

- **Build-time vs. runtime split:**
  - **Build-time:** Claude Code agents/skills + superpowers is the development harness that
    writes the code. Not submitted. Not part of the runtime.
  - **Runtime:** a Google ADK multi-agent content pipeline + an MCP server + the Google
    Agents CLI make up the agentic system that produces and serves content.
- **No generative AI in the end-user experience (MVP).** All content — text and narration
  audio — is pre-generated by the pipeline and served as static files. The only runtime AI is
  **non-generative**: a local embedding model powers semantic matchmaking on the homepage
  (§5.1, Rung 2). The web runtime needs no Gemini key at serve time. (The generative "miss
  sketch" is deferred post-MVP — §5.1.)

## 3. Target user

Music-curious Gen-X nerds and educators using music to teach history and culture. They value
depth and discovery. They want to feel connected to roots and musical traditions and connecting threads —
a quiet rebellion against the homogenising effect of globalisation.

## 4. Goals & non-goals

### Goals (MVP)
1. A working, visually rich web app for exploring music Journeys.
2. A seeded library of **≈10 Journeys**, each with 6–10 songs, each song with a real preview snippet.
3. All content produced by an **ADK multi-agent content pipeline**.
4. Immediate listening via **30-second preview snippets** (iTunes/Deezer, no auth required).
5. **Text-to-speech narration** for every journey intro and every song blurb — pre-generated
   by the pipeline using Gemini TTS, served as static audio files. The app functions as a
   curated library of listenable music history podcasts.
6. Downloadable playlist at the end of each Journey as a snippet mixtape + streaming deep links.
7. API-first, headless, modular Python core; the web app is the first of many possible clients.

### Non-goals (explicitly out of scope for MVP)
- Geolocation / geofencing / push notifications.
- Obscure / endangered-culture content.
- Content verification at scale / community moderation.
- Streaming-account integration (e.g., auto-creating Spotify playlists).
- User accounts, login, cross-session persistence.
- Full-track audio hosting or downloads (only 30s snippets + deep links).

## 5. Core experience & user flows

### 5.1 Homepage & discovery

The homepage is a **grid of Journey cards** (title, theme tag, thumbnail) plus an
**"I'm feeling lucky"** button that opens a random Journey.

Discovery is **matchmaking, not search**: with a small curated library, the job is to take a
vague human wish and point it at the right Journey — or honestly admit we don't have it yet.
Three mechanisms, in increasing sophistication:

#### Rung 1 — Evocative starter prompts (no AI)
A row of hand-written prompt chips beneath the search box, e.g.
*"Take me somewhere political" · "A story that moves across a map" · "One object, many decades" ·
"Surprise me"*. Each chip maps to a specific Journey (or to "I'm feeling lucky"). Zero cost,
instant, and teaches users what kind of input the box understands. Hand-curated in config.

#### Rung 2 — Semantic matchmaking (AI at startup, not on-the-fly)
A free-text box that understands intent, not just keywords. So *"songs about money and class"*
finds the shoes Journey via *Diamonds on the Soles of Her Shoes*, even though it never says
"money."
- Each Journey's `title` + `theme` + `blurb` is embedded **once at startup** with a small
  local sentence-embedding model (e.g. `sentence-transformers` MiniLM, 384-dim, CPU).
- At query time, embed the user's text (~10ms) and cosine-match against the Journey vectors
  held in memory; return the best match above a confidence threshold.
- **No network, no API key, no per-query cost, and no hallucination** — it can only ever
  return a Journey that already exists. The only cost is one local model dependency.

#### Rung 3 — The honest miss
When the top match falls below the confidence threshold (*"songs about dinosaurs"*), don't
fake it. Show: *"No journey about that yet — the closest we have is X,"* plus a one-click
**"want this? +1"** that logs demand to seed the content roadmap. Honesty as a feature.

> **Deferred (post-MVP):** a "miss sketch" — on a miss, a single short Gemini call writes a
> 2-sentence *sketch* of what such a journey might explore, labeled "a sketch, not a real
> journey yet." Decided **out of MVP scope** to keep the web runtime fully non-generative (no
> Gemini key at serve time, no extra failure path). Architecture should leave room for it.

### 5.2 Journey page

1. **Journey header** — title, subtitle, theme tag.
2. **Intro narration** — the journey blurb plays automatically as a TTS audio clip on page
   load. Text is visible alongside so users can read while listening (or read if audio is off).
3. **Song list** — displayed in chronological order. Songs ahead of the current step are
   shown as collapsed placeholders (title + artist only). The current step is fully open.
4. A **progress indicator** shows position in the journey (e.g., "Song 3 of 8").
5. User steps through one song at a time; the "next song" control only appears after the
   current song's audio pair (narration + preview) has started playing or the user skips.
6. A **"Next journey →"** control is persistently visible (header or sidebar) so a user can
   bail early if the theme doesn't suit them. Uses the same selection logic as the end-of-journey
   card (§5.5 — same-theme group first, random otherwise).

### 5.3 Song step — the podcast unit

Each song step is a self-contained podcast unit:

1. **Narration plays** — the song blurb TTS audio auto-plays. Text visible alongside.
2. **Preview plays** — after narration ends (or user skips narration), the 30-second
   iTunes/Deezer snippet auto-plays.
3. **Advance** — "Next →" button appears. User taps to unlock the next step.

The audio sequence per step is: *narration clip → song preview clip → advance*.
Users can skip either clip individually. The blurb text is always visible regardless of
audio state, so the experience works without sound.

Visible metadata per step: title, artist, `month_year`, `place` (where recorded).

### 5.4 End of journey

After the final song:
- A **closing narration** — TTS audio of a short paragraph tying the arc together,
  auto-playing after the last song preview ends.
- A **downloadable playlist** — a file (or copyable list) of all song + artist pairs with
  deep links to open in the user's preferred streaming service.
- A **"try another journey"** card pointing to the next journey resolved by the same-theme-first,
  then-random logic described in §5.5.

### 5.5 Navigation

- Session-only breadcrumb shows which journey the user is in.
- No accounts, no cross-session state.

#### Journey switching (mid-journey)

A **"Next journey →"** control is visible at all times while inside a journey — not just
at the end. This lets a user bail early if the theme doesn't suit them without returning
to the homepage.

**Selection logic:**

1. If the current journey's `theme` matches the `theme` of one or more other journeys in the
   library, the next is drawn from that same-theme group (round-robin or random within the group).
2. If no other journey shares the theme, a random other journey is selected.

The "try another journey" card at the end of a journey (§5.4) uses the same selection logic.
The control at the top/side of a journey page uses the same route: `GET /api/journeys/next?from={id}`
returns the resolved target journey, which the server then redirects to. No client-side
selection logic.

## 6. Proof of Concept (vertical slice — build this first)

Prove the entire path end-to-end with **two hand-seeded Journeys** before building the
pipeline or scaling content. Two examples are chosen deliberately to cover both journey
archetypes — one where the thread is **historical causation**, one where it is **associative**.

The POC must also demonstrate the **discovery/selection experience** (§5.1), not just the
Journey reading view. At minimum: the homepage grid, the starter prompt chips, and arriving at
a Journey *by selecting it* (via a chip or the search box) rather than by direct URL. The
Great Migration journey is reachable through the **"Take me somewhere political"** chip; the
shoes journey through the **"One object, many decades"** chip.

### POC Journey A — causal/historical: "From the Delta to Detroit"

Six songs, 1936–1982, tracing how the Great Migration physically moved Black music from
Mississippi through Chicago and Detroit to the South Bronx. Each song blurb explicitly
references the geographic and sonic thread of the previous entry — the connection is *lineage*.

| # | Title | Artist | Year | Place |
|---|-------|--------|------|-------|
| 1 | Cross Road Blues | Robert Johnson | 1936 | San Antonio, TX |
| 2 | Hoochie Coochie Man | Muddy Waters | 1954 | Chicago, IL |
| 3 | Johnny B. Goode | Chuck Berry | 1958 | Chicago, IL |
| 4 | My Girl | The Temptations | 1964 | Detroit, MI |
| 5 | Respect | Aretha Franklin | 1967 | New York City, NY |
| 6 | The Message | Grandmaster Flash | 1982 | South Bronx, NY |

### POC Journey B — associative/topical: "Walk a Mile in These"

Eight songs, 1955–1993, all built around a single object — the shoe. The thread is *not*
causal: Carl Perkins did not lead to Nancy Sinatra. Instead each blurb compares what its
artist *does* with the shared image, tracing a conceptual arc from shoe-as-literal-object to
shoe-as-metaphor. This is the "songs about cars / songs by siblings" archetype.

| # | Title | Artist | Year | Place |
|---|-------|--------|------|-------|
| 1 | Blue Suede Shoes | Carl Perkins | 1955 | Memphis, TN |
| 2 | These Boots Are Made for Walkin' | Nancy Sinatra | 1965 | Los Angeles, CA |
| 3 | Boogie Shoes | KC and the Sunshine Band | 1975 | Hialeah, FL |
| 4 | The Angels Wanna Wear My Red Shoes | Elvis Costello | 1977 | London, England |
| 5 | Goody Two Shoes | Adam Ant | 1982 | London, England |
| 6 | My Adidas | Run-DMC | 1986 | New York City, NY |
| 7 | Diamonds on the Soles of Her Shoes | Paul Simon | 1986 | New York City, NY |
| 8 | Walking in My Shoes | Depeche Mode | 1993 | Madrid, Spain |

### What the POC proves
- Full vertical slice: discovery → Journey page → narrated song steps → snippet playback →
  end-of-journey playlist download.
- That a user can **find and choose** a Journey through the selection experience, not just
  land on one by URL.
- That the podcast unit (narration → preview → advance) works end-to-end for at least one song.
- That the Journey data shape holds for **both** archetypes without schema changes.

### What the POC may stub
- POC data is hand-seeded; the ADK pipeline is built and validated separately (M4).
- TTS audio for the POC can be hand-generated (one-off Gemini TTS call) to prove the UX
  without needing the full pipeline. The pipeline generates it automatically from M4 onward.
- Visual polish and image assets can be minimal.

## 7. Content model

### Journey

| Field | Type | Notes |
|-------|------|-------|
| `id` | string | slug, e.g. `journey:great-migration` |
| `title` | string | display name |
| `subtitle` | string | one-line description |
| `theme` | string | tag, e.g. "geopolitical movement", "siblings", "cars" |
| `blurb` | string | the "why these songs together" narrative (1–3 paragraphs) |
| `intro_audio_url` | string | TTS narration of `blurb` — pre-generated by pipeline, served as static `.mp3` |
| `outro_audio_url` | string | TTS narration of closing paragraph — pre-generated by pipeline |
| `image_url` | string | representative thumbnail |
| `prompt_chips` | string[] | homepage chip labels that route to this journey |
| `songs` | Song[] | ordered list, 6–10 entries |

### Song (within a Journey)

| Field | Type | Notes |
|-------|------|-------|
| `position` | int | 1-indexed order in the journey |
| `title` | string | song title |
| `artist` | string | performing artist(s) |
| `month_year` | string | e.g. `"November 1936"` or `"1954"` — string, not a date type |
| `place` | string | where the song was **recorded** (not artist origin, not lyrical setting) |
| `blurb` | string | how this song ties into the journey and connects to prior songs |
| `blurb_audio_url` | string | TTS narration of `blurb` — pre-generated by pipeline, served as static `.mp3` |
| `preview_url` | string | 30-second snippet from iTunes/Deezer — **from MCP tool only, never LLM-generated** |
| `image_url` | string | album art URL |
| `streaming_links` | object | `{spotify, apple_music, youtube}` — deep links |

## 8. Theme taxonomy (non-exhaustive)

Journey themes are open-ended, but seed examples include:

- **Same topic** — songs about a shared subject (cars, rivers, mothers, war)
- **Sibling artists** — songs recorded by siblings (Everly Brothers, Bee Gees, Gallagher brothers)
- **Geopolitical movement** — songs that document or were shaped by a historical event/movement
- **One city, one decade** — a snapshot of a specific place and time
- **Genre birth** — the songs that defined a new genre, in the order they appeared
- **Dialogue across time** — songs in conversation with each other (covers, responses, samples)
- **Migration arcs** — music that followed people across borders or continents

## 9. Content-generation pipeline (ADK multi-agent, runtime)

A Google ADK multi-agent system that populates the content store. Agents:

| Agent | Role |
|-------|------|
| **Orchestrator** | Deterministic workflow agent (code, not LLM). Drives a run for a given theme seed. |
| **Journey researcher** | LlmAgent. Produces journey `blurb`, selects 6–10 songs with `title`, `artist`, `month_year`, `place`, per-song `blurb`, and closing paragraph. |
| **Song fetcher** | LlmAgent + MCP. Resolves `preview_url` + `image_url` + `streaming_links` for each song via iTunes/Deezer. |
| **Fact checker** | LlmAgent + MCP. Verifies that `month_year` and `place` (where recorded) are correct against MusicBrainz/Wikipedia. Flags unverifiable songs. |
| **Narration generator** | Calls Gemini TTS API for journey intro, each song blurb, and closing paragraph. Saves `.mp3` files to `data/audio/{journey_id}/`. Populates `intro_audio_url`, `outro_audio_url`, `blurb_audio_url`. |
| **Graph linker** | Persists the fully-enriched journey (text + audio URLs) to the content store. Idempotent upserts. |

Each agent has:
- A skill spec under `docs/agents/`
- A trajectory `.evalset` test (expected tool-call path + output schema validation) to reduce hallucination
- Independent validation before being wired into the orchestrator

## 10. MCP server (external sources)

Wraps external sources as MCP tools — two families:

- **Music lookups:** iTunes Search API (no auth) and/or Deezer — song → `preview_url`,
  `image_url`, metadata.
- **Verification lookups:** Wikipedia/Wikidata, MusicBrainz — verify recording date, recording
  location, artist identity.

`place` (where recorded) must come from MusicBrainz, not from the LLM.
`preview_url` must come from an iTunes/Deezer tool result, never LLM-generated.

## 11. Audio & playlists

### TTS narration (new, must-have)
- **Model:** Gemini TTS (`gemini-2.5-flash-preview-tts`) via the same `GEMINI_API_KEY`.
- **Generated at pipeline time** — never on-demand at serve time. Audio files stored in
  `data/audio/{journey_id}/` and served by FastAPI as static files at `/audio/...`.
- **What gets narrated:**
  - Journey intro (`intro_audio_url`) — the journey-level blurb read aloud.
  - Per-song blurb (`blurb_audio_url`) — the contextual story for each song, read before
    the preview plays.
  - Closing paragraph (`outro_audio_url`) — the end-of-journey wrap-up.
- **Format:** `.mp3`, mono, consistent voice across all content in a journey.
- **Fallback:** if a TTS call fails, the field is `null` and the UI shows the text-only
  version. The experience degrades gracefully — text is always visible.

### Song previews
- 30-second snippets from iTunes/Deezer, fetched at pipeline time, stored on the Song record.
  Mainstream catalog only at MVP.

### End-of-journey playlist
- Downloadable as plain text (title + artist + streaming deep links) + `.m3u` variant.
- No full-track hosting, no audio file downloads.

## 12. Architecture (high level)

- **API-first / headless:** all logic behind a clean, modular Python API (FastAPI).
- **Frontend:** clean, visual web UI — album art, journey cards, song step layout. Desktop +
  mobile browser.
- **Stateless:** session-only; no accounts, no persistence beyond the data store.
- Specific framework / DB / deployment choices are deferred to PLANNING.md.

## 13. Phase-2 directions (architecture should accommodate, not implement)

- Community edits — label AI-generated content, allow corrections.
- Streaming-account linking — auto-create playlists.
- User accounts, saved journeys, listening history.
- Native mobile app.

## 15. Decisions locked (carry into PLANNING.md)

- **Pipeline role — "human picks theme, AI builds":** the human supplies a theme; the
  `journey_researcher` agent selects the 6–10 songs and writes all blurbs; `song_fetcher` +
  `fact_checker` enrich and verify; a **human review gate** approves before publish.
  `journey_researcher` is the heart of the pipeline.
- **Web runtime is non-generative (MVP):** Rung-3 "miss sketch" is **out of scope**; no Gemini
  key needed at serve time. Architecture leaves room to add it later.
- **Storage:** flat JSON (`journeys.json`) behind the repository interface (swap to SQLite later).
- **Step-unlock UX:** server-rendered HTMX swap, not client-side JS state.
- **Playlist download:** plain-text list with streaming deep links, plus an `.m3u` variant.
- **Rung-2 embedding:** `sentence-transformers` MiniLM (384-dim, CPU), loaded once at startup;
  cosine "miss" threshold tuned empirically during build.
